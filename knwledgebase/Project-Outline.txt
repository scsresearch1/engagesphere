1) PROJECT DEFINITION

1.1 Purpose
  - EngageSphere is a web application combining a social-style feed (posts, comments,
    likes, notifications) with an analytics-heavy dashboard and an admin console.
  - Target users: Students and Faculty as primary participants; Administrators for
    governance (approvals + moderation).

1.2 Current delivery state (as implemented)
  - Phase A (local): Full UI with persistence in browser localStorage when Firebase
    web env vars are NOT configured.
  - Phase B (connected): Firebase Realtime Database (RTDB) at node engageApp/ syncs
    the same logical data model; granular reads/writes + onValue listeners.
  - Self-serve registration creates Student or Faculty accounts in a PENDING state;
    an Admin must approve before first login. Admin accounts are NOT creatable via
    public signup (seed / manual provisioning only).
  - Firebase Authentication and Cloud Firestore are NOT yet wired into the client;
    passwords remain on user documents for demo (NOT production practice).
  - Hosting target: Netlify (netlify.toml + dist publish); SPA fallback configured.

1.3 Roles
  - Student, Faculty, Admin (TypeScript union Role in src/types.ts).
  - SelfServeRole = Student | Faculty (signup constraint).
  - Admin-only route: /admin (PrivateRoute allowRoles={["Admin"]}).

5) DATA MODEL (logical)

Stored under AppData / engageApp:
  users[]           id, name, email, password, role, bio, profileImage, joinedAt,
                    optional approvalStatus: "pending"
  posts[]           id, userId, content, optional imageUrl, createdAt, optional updatedAt
  comments[]        id, postId, userId, text, createdAt
  likes[]           id, postId, userId, createdAt
  notifications[]   id, receiverUserId, senderUserId, type like|comment, postId,
                    message, isRead, createdAt

RTDB physical layout: engageApp/{users|posts|comments|likes|notifications}/{id}

6) FUNCTIONAL MODULES (implementation mapping)

6.1 Authentication & session
  - Files: AppStateContext (login, logout), LoginPage, PrivateRoute, AuthRoute.
  - Session: localStorage key engagesphere-current-user-v1.
  - appReady: avoids login redirect before first RTDB snapshot.

6.2 Registration & governance
  - SignupPage: Student|Faculty only; success screen without auto-login.
  - register(): sets approvalStatus pending; Admin role rejected at type level + copy.
  - AdminPage: approvePendingUser, rejectPendingUser; stats tile for pending count.

6.3 Feed & posts
  - FeedPage, PostDetailsPage, PostComposer, PostCard.
  - createPost, updatePost, deletePost (cascade) in context + rtdbWrites.

6.4 Comments & likes
  - addComment, deleteComment, toggleLike; notifications to post owner.

6.5 Notifications
  - NotificationsPage + markAllNotificationsRead; sidebar badge unreadCount.

6.6 Profile
  - ProfilePage; updateProfile patches user fields.

6.7 Dashboard & analytics
  - DashboardPage orchestrates widgets; useDashboardDerived computes KPIs, charts,
    timeline, leaderboard, heat week, streak, role pie, notification donut, etc.
  - DashboardGlobalSearch: query, role filter, sort, date range; trending tags filter
    by substring on post content (fixed tag vocabulary).

6.8 Administration
  - AdminPage: registration approvals, user directory, post list with remove,
    recent comments with remove.

6.9 Persistence modes
  - Local: STORAGE_KEY engagesphere-static-state-v1 JSON.
  - RTDB: onValue(engageApp); mutations via rtdb* helpers.

6.10 Seeding & ops
  - npm run seed:rtdb — requires FIREBASE_SERVICE_ACCOUNT_PATH, optional
    FIREBASE_DATABASE_URL (default engagesphere-a06c3 RTDB URL in script).
  - resetDemoToSeed() from login: full tree replace + logout.

--------------------------------------------------------------------------------

7) SECURITY NOTES (current vs ideal)

  - database.rules.json (dev): public read/write under engageApp — must be replaced
    before any real user data with auth-based rules.
  - Demo passwords in cleartext in database — acceptable only for class demo;
    production requires Firebase Auth + server-side or rule-protected profiles.
  - Service account JSON must never ship in frontend or public repos.
